ATM TOC GUIDE

Formal Languages and Automata Used in the Project

Project Overview

This project models an ATM transaction system using concepts from Theory of Computation. The base flow starts from card insertion, moves to PIN verification, then operation selection, withdrawal, and exit. We first represented this as a DFA and then extended the same system using NFA, ε-NFA, Regular Expressions, Context-Free Grammar, Pushdown Automata, and a simple Turing Machine.

Initially, the project implemented a DFA for ATM operations. We extended it to include NFA and ε-NFA for non-deterministic behavior, demonstrated NFA to DFA conversion, represented the system using Regular Expressions and Context-Free Grammar, and further modeled computation using PDA and a basic Turing Machine.

1. DFA

A Deterministic Finite Automaton has exactly one transition for each valid input from a state. In this ATM project, the main simulator is a DFA because the path is fixed when the user behaves correctly.

q0 --card--> q1
q1 --pin(correct)--> q2
q2 --op--> q3
q3 --withdraw--> q4
q4 --exit--> q0

2. NFA

An NFA allows multiple possible next states for the same input. We used this idea to explain ATM branching behavior such as PIN acceptance versus security failure.

q1 --pin--> {q2, q_dead}
q2 --op--> {q3, q_dead}

3. ε-NFA

An ε-NFA includes epsilon transitions, which move the automaton without consuming any input. In the ATM model, this is used to explain automatic internal movement between stages.

q1 --ε--> q2

4. NFA to DFA Conversion

We used subset construction to convert multiple NFA states into a single DFA state. This removes non-determinism while preserving accepted strings.

{q1, q2} becomes one DFA state
{q2, q_dead} becomes another DFA state

5. Regular Expression

The ATM workflow can be written as a regular expression describing the valid sequence of symbols.

(card)(pin)(op)(withdraw)(exit)

In the project, we added a regex checker to test whether a user-entered sequence matches this exact pattern.

6. Context-Free Grammar

CFG represents the ATM flow as production rules. This shows the same language from a grammar perspective rather than only a state-transition perspective.

S → card A
A → pin B
B → op C
C → withdraw D
D → exit

7. PDA

A Pushdown Automaton uses a stack. We used stack operations to explain memory-like behavior in the ATM, such as recording session events like card insertion and PIN entry.

Input: card  => push CARD
Input: pin   => push PIN
Input: exit  => pop stack

8. Turing Machine

The Turing Machine is the most general computation model. We represented ATM inputs on a tape and moved the head one step to the right after each read.

[card] [pin] [op] [withdraw] [exit]
  ^
Head reads and moves right

How We Used TOC in the Project

DFA was used for the main ATM transaction simulator. NFA and ε-NFA were added to explain branching and automatic transitions. NFA to DFA conversion was added to show formal conversion of non-deterministic behavior into deterministic behavior. Regular Expression and CFG were used to describe the ATM language formally. PDA was used to simulate stack memory, and a Turing Machine section was added to show tape-based computation.

This makes the project strong both technically and academically because it does not only show one automata model. It connects multiple TOC topics to one real-life system in a structured and easy-to-explain way.
