ATM TOC GUIDE

Formal Languages and Automata Used in the Project

Project Overview

This project models an ATM transaction system using concepts from Theory of Computation. The base flow starts from card insertion, moves to PIN verification, then operation selection, withdrawal, and exit. We first represented this as a DFA and then extended the same system using NFA, ε-NFA, Regular Expressions, Context-Free Grammar, Pushdown Automata, a simple Turing Machine, lock-state control, timeout recovery, transaction logging, and DFA minimization.

Initially, the project implemented a DFA for ATM operations. We then added a lock state for repeated PIN failures, inactivity timeout reset, timestamped transition logging, NFA and ε-NFA behavior, NFA to DFA conversion, DFA minimization, Regular Expressions, Context-Free Grammar, PDA, and a basic Turing Machine.

1. DFA

A Deterministic Finite Automaton has exactly one transition for each valid input from a state. In this ATM project, the main simulator is a DFA because the path is fixed when the user behaves correctly.

q0 --card--> q1
q1 --pin(correct)--> q2
q2 --op--> q3
q3 --withdraw--> q4
q4 --exit--> q0

The simulator also keeps a trap state q_dead for protocol violations and a lock state q_lock for repeated PIN failures.

2. NFA

An NFA allows multiple possible next states for the same input. We used this idea to explain ATM branching behavior such as PIN acceptance, retry, lock, or security failure.

q1 --pin--> {q2, q1, q_lock, q_dead}
q2 --op--> {q3, q_dead}

3. ε-NFA

An ε-NFA includes epsilon transitions, which move the automaton without consuming any input. In the ATM model, this is used to explain automatic internal movement between stages.

q1 --ε--> q2

4. NFA to DFA Conversion

We used subset construction to convert multiple NFA states into a single DFA state. This removes non-determinism while preserving accepted strings.

{q1, q2} becomes one DFA state
{q1, q_lock} becomes another DFA state
{q2, q_dead} becomes another DFA state

5. DFA Minimization

After building the DFA, we apply partition refinement to study whether any states can be merged. This demonstrates formal optimization of the automaton.

Initial partition: {q0} and {q1, q2, q3, q4, q_lock, q_dead}
Refine by transition behavior
Keep q_lock separate from q_dead because q_lock can recover

6. Regular Expression

The ATM workflow can be written as a regular expression describing the valid sequence of symbols.

(card)(pin)(op)(withdraw)(exit)

In the project, we added a regex checker to test whether a user-entered sequence matches this exact pattern.

7. Context-Free Grammar

CFG represents the ATM flow as production rules. This shows the same language from a grammar perspective rather than only a state-transition perspective.

S -> card A
A -> pin B
B -> op C
C -> withdraw D
D -> exit

8. PDA

A Pushdown Automaton uses a stack. We used stack operations to explain memory-like behavior in the ATM, such as recording session events like card insertion and PIN entry.

Input: card  => push CARD
Input: pin   => push PIN
Input: exit  => pop stack

9. Turing Machine

The Turing Machine is the most general computation model. We represented ATM inputs on a tape and moved the head one step to the right after each read.

[card] [pin] [op] [withdraw] [exit]
  ^
Head reads and moves right

10. Security and Recovery Extensions

The updated simulator also demonstrates industrial workflow controls. These features help connect the formal automaton with practical ATM safety rules.

3 wrong PIN attempts  -> q_lock
No input for 30 sec   -> q0
Each transition       -> timestamped log entry

How We Used TOC in the Project

DFA was used for the main ATM transaction simulator. NFA and ε-NFA were added to explain branching and automatic transitions. NFA to DFA conversion and DFA minimization were added to show formal construction and optimization. Regular Expression and CFG were used to describe the ATM language formally. PDA was used to simulate stack memory, and a Turing Machine section was added to show tape-based computation.

The project also includes practical security extensions such as a lock state after three incorrect PIN attempts, automatic timeout reset to q0, and timestamped transition logging. This makes the project strong both technically and academically because it connects automata theory with real ATM control rules in a structured and easy-to-explain way.
