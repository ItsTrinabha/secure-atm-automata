How A User Approaches This Project

1. Entry Point

The user first lands on the main ATM Automata dashboard. The default tab is Console & FSM, which acts as the interactive starting point of the project.

2. Initial State

The ATM screen begins at q0: IDLE with the message WELCOME / INSERT CARD. This tells the user the machine is waiting for the first valid symbol.

3. Card Insertion

The user clicks Insert Card. The machine transitions from q0 to q1, where authentication begins and the PIN input field becomes active.

4. PIN Validation

The user enters the PIN. If the value is 1234, the machine moves to q2. If the PIN is wrong, the attempt counter increases and repeated failure eventually leads to q_lock.

5. Operation Selection

After authentication, the user clicks Choose Op, which moves the machine from q2 to q3. This represents choosing a banking operation in the formal workflow.

6. Withdrawal Stage

The user clicks Withdraw, which transitions the system from q3 to q4. This is the successful transaction-processing stage.

7. Exit and Reset

The user clicks Cancel/Exit and the machine returns to q0. This completes one accepted transaction string and returns the ATM to idle.

8. Continuous Feedback

During every step, the project updates the ATM screen, state graph, transition matrix, timeout counter, attempt tracker, and transaction logs so the user can see both system behavior and formal automata behavior together.

Normal Transaction Flow

Accepted Sequence

q0 --card--> q1
q1 --pin(correct)--> q2
q2 --op--> q3
q3 --withdraw--> q4
q4 --exit--> q0

This is the ideal user path. It corresponds to the accepted ATM language and shows how the user completes a valid transaction from start to finish.

Error and Security Flow

When The User Makes Mistakes

If the user presses buttons in the wrong order, the system moves to q_dead. If the user enters the wrong PIN three times, the machine moves to q_lock. In both cases, the user must use Hardware Reset to start again.

wrong sequence -> q_dead
3 wrong PINs   -> q_lock
timeout        -> q0
All User Cases In This Simulator

Case 1. Successful Transaction

The user follows the correct order: Insert Card, correct PIN 1234, Choose Op, Withdraw, and Cancel/Exit. The machine moves through q0 -> q1 -> q2 -> q3 -> q4 -> q0 and the transaction is accepted. In Theory of Computation terms, this is the main DFA path because each valid input has one exact next state. It also matches the project language and the regular expression (card)(pin)(op)(withdraw)(exit).

δ(q0, card) = q1
δ(q1, pin) = q2
δ(q2, op) = q3
δ(q3, withdraw) = q4
δ(q4, exit) = q0

Case 2. One Wrong PIN

If the user enters a wrong PIN once, the ATM stays in q1. The attempt counter increases, the screen shows an authentication failure message, and the user gets another chance to enter the PIN. This is useful for explaining NFA thinking, because the input symbol pin can conceptually lead to more than one possible result such as retry, success, lock, or rejection depending on the condition.

δ(q1, wrong_pin) = q1
δN(q1, pin) = {q2, q1, q_lock, q_dead}

Case 3. Three Wrong PINs

If the user enters the wrong PIN three times, the system moves to q_lock. This represents a security lock state where the normal flow stops and the user cannot continue the transaction until the machine is reset. In formal language terms, this path is outside the accepted language, so the computation does not complete the valid ATM string.

δ(q1, wrong_pin^3) = q_lock
w ¬in L(M)

Case 4. Wrong Button Order

If the user presses buttons in an invalid sequence, such as trying to withdraw before authentication or choosing an operation at the wrong time, the machine moves to q_dead. This is the trap state for protocol violations. It helps explain how a DFA rejects strings that do not follow the required order and how invalid derivations fail to match the grammar of the transaction process.

δ(q0, withdraw) = q_dead
δ(q1, op) = q_dead
invalid string → reject

Case 5. Cancel Or Exit Midway

If the user presses Cancel/Exit from a valid intermediate state like q1, q2, q3, or q4, the ATM returns to q0. This models a safe session termination without completing the full transaction. In automata terms, the machine returns to the start state, but the partial input string is still not part of the accepted final transaction pattern.

δ(q1, exit) = q0
δ(q2, exit) = q0
δ(q3, exit) = q0

Case 6. Timeout

If the user does nothing for the session timeout period, the simulator automatically resets the session back to q0. This shows how an ATM protects itself by ending inactive sessions. This case is also useful for discussing internal system movement similar to an automatic transition, which connects nicely to the idea of epsilon-style movement in extended automata explanations.

δ(qi, timeout) = q0
q1 --ε--> q2  (extended explanation)

Case 7. Hardware Reset

If the user reaches q_lock or q_dead, the normal buttons are no longer enough. The user must press Hardware Reset, which reboots the simulator, clears counters, and returns the machine to q0. This restart helps show that after rejection or lockout, a fresh computation begins again from the initial state.

δ(q_lock, reset) = q0
δ(q_dead, reset) = q0

Case 8. Live System Feedback

In every case, the user can observe the ATM display, transition equation, graph highlight, attempt counter, timeout counter, and log messages. This helps the user understand both the practical ATM behavior and the formal state-machine behavior at the same time. The state transitions illustrate DFA logic, the alternative outcomes around PIN handling help explain NFA, the accepted string matches a regular expression, and the ordered workflow can also be described with a simple CFG such as S -> card A -> pin B -> op C -> withdraw D -> exit.

M = (Q, Σ, δ, q0, F)
Q = {q0, q1, q2, q3, q4, q_lock, q_dead}
Σ = {card, pin, op, withdraw, exit}
F = {q0}
